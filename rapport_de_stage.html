II. Présentation du projet de stage

2.1 Contexte et problématique

La gestion des Assemblées Générales (AG) est un processus fondamental dans la vie des entreprises cotées et des sociétés anonymes. Elle implique la convocation des actionnaires, la présentation de résolutions soumises au vote, et la collecte formelle des suffrages.

Au sein d'Attijariwafa Bank, ce processus nécessitait une solution numérique sécurisée permettant de :

  • Gérer les entreprises et leurs actionnaires de manière centralisée
  • Organiser les assemblées générales (date, lieu, ordre du jour)
  • Permettre aux actionnaires de voter à distance sur les résolutions
  • Assurer la traçabilité et la sécurité de chaque vote exprimé
🎯 Mission du stagiaire

Concevoir et développer une application web full stack permettant la gestion complète des assemblées générales et le vote électronique sécurisé des actionnaires, avec interfaces distinctes pour l'administrateur et les actionnaires.

IV. Réalisation Technique — Backend

4.1 Structure du projet Spring Boot

Le backend suit le patron d'architecture MVC enrichi (Model-View-Controller) avec une séparation en 5 couches distinctes :

  • config/ — Configuration globale (Sécurité, CORS, initialisation des données)
  • controller/ — Points d'entrée REST (reçoivent et retournent du JSON)
  • service/ — Logique métier (règles de gestion, orchestration)
  • repository/ — Accès base de données (Spring Data JPA)
  • entity/ — Modèles JPA mappés aux tables SQL
  • dto/ — Objets de transfert de données (découplage API/BDD)
  • security/ — Authentification JWT (filtre, utilitaire, UserDetails)

4.2 Sécurité — Authentification JWT

La sécurité de l'API repose sur le standard JSON Web Token (JWT). Chaque token contient le nom d'utilisateur, son rôle, la date d'émission et la date d'expiration. La signature HMAC-SHA256 garantit l'intégrité du token.

🔑 Cycle d'authentification

Connexion → Vérification BCrypt → Génération JWT (validité 24h) → Token envoyé au client
Requête suivante → Header Authorization: Bearer {token} → JwtAuthFilter valide → Accès autorisé

Endpoint Méthode Accès Description
/api/auth/login POST Public Connexion, retourne JWT + profil
/api/auth/register POST Public Inscription nouvel utilisateur
/api/admin/** Tous ADMIN uniquement CRUD complet toutes entités
/api/user/ags GET USER uniquement AGs de l'actionnaire connecté
/api/user/vote POST USER uniquement Exprimer un vote
/api/user/votes GET USER uniquement Historique votes personnels

4.3 Logique de vote — Contrainte d'unicité

Le service de vote implémente une règle métier essentielle : un actionnaire ne peut voter qu'une seule fois par résolution. Si un vote existe déjà, il est mis à jour (changement de choix) ; sinon, un nouveau vote est créé. Le poids du vote correspond au nombre d'actions de l'actionnaire. 

// Logique dans VoteService.java
Vote vote = voteRepository
    .findByActionnaireAndResolution(actionnaire, resolution)
    .orElse(new Vote());  // UPDATE si existe, INSERT sinon

vote.setChoix(request.getChoix());     // POUR / CONTRE / NEUTRE
vote.setPoids(actionnaire.getNombreActions()); // Poids = nb actions
voteRepository.save(vote);

5.4 Interface utilisateur — Pages principales

Page de connexion (LoginPage.jsx)

Double onglet Connexion / Inscription. Après connexion, redirection automatique selon le rôle : ADMIN → gestion des entreprises, USER → tableau de bord.

Page de vote (AGVotePage.jsx)

Liste des assemblées générales de l'actionnaire en accordéon. Pour chaque résolution, trois boutons de vote (POUR en vert, CONTRE en rouge, ABSTENTION en gris). Le vote déjà exprimé est mis en surbrillance. Feedback visuel immédiat.

Tableau de bord (DashboardPage.jsx)

Statistiques personnelles : nombre total d'AG, AGs à venir, AGs passées, résolutions totales. Les AGs sont automatiquement classées selon leur date.
